# Spec Governance

**Status:** Draft
**Last Updated:** 2026-03-18
**Surfaces:** All
**Code:** System-wide

---

## Overview

The PPDS specs directory has grown to 32 files with no governing philosophy. Feature specs (living — "what Plugin Traces should be") and project specs (ephemeral — "build 6 panels for parity") are mixed together. Surface-specific duplicates exist (`plugin-traces.md` AND `tui-plugin-traces.md`). Enhancement specs exist for implementation details (`vscode-data-explorer-monaco-editor.md`). The README index lists only half the specs on disk. No guidance exists for naming, lifecycle, granularity, or what happens after a spec reaches "Implemented."

This spec establishes governance rules, updates the template, adds a reconciliation capability to `/spec`, and provides restructuring guidelines for the existing 32 specs.

### Goals

- **Clear rules**: Constitution amendment defines naming, lifecycle, and granularity — loaded by every skill automatically
- **Living specs**: Specs evolve with the feature, not frozen after initial implementation
- **Honest index**: README auto-generated from frontmatter, impossible to drift
- **Code-to-spec reconciliation**: Enhanced `/spec` reads code and updates specs, closing the feedback loop
- **Restructuring plan**: Guidelines for session 2 to clean up the existing 32 specs

### Non-Goals

- Restructuring the actual specs (session 2, same branch)
- Workflow-enforcement.md restructure (separate concern, separate branch)
- Spec archival/deprecation policy (pre-v1: deleted = gone)
- Automated spec generation from code without human review
- Changes to the workflow pipeline hooks or workflow state schema

## Specification

### Constitution Amendment — Spec Laws

Add the following section to `specs/CONSTITUTION.md` after the existing Resource Laws:

```markdown
## Spec Laws

**SL1.** One spec per domain concept, named after the thing — not the project, surface, or
enhancement. `plugin-traces.md` not `tui-plugin-traces.md`. `data-explorer.md` not
`vscode-data-explorer-monaco-editor.md`. Cross-cutting architectural patterns are legitimate
standalone specs.

**SL2.** Specs are living documents — updated as the feature evolves. Plans (`.plans/`) are
ephemeral and consumed by implementation. Project coordination documents (parity, polish, audit)
are plans, not specs.

**SL3.** Surface-specific behavior (TUI screen layout, Extension panel wiring, MCP tool schema)
lives in surface sections within the domain spec, not in separate spec files.

**SL4.** Spec frontmatter `**Code:**` must contain grep-friendly path prefixes, not prose like
"Multiple (see spec)". Every spec with an implementation must have at least one code path.
System-wide specs (architecture, governance) use `**Code:** System-wide`.

**SL5.** Specs for removed features are deleted. No archival, no deprecation ceremony, no
tombstones.
```

### Spec Template Changes

**Frontmatter updates:**

1. Replace `**Version:** 1.0` with nothing — version numbers are removed from the template. Version tracking moves to the Changelog section.

2. Add `**Surfaces:**` line after `**Code:**`:
   ```
   **Surfaces:** CLI | TUI | Extension | MCP | All | N/A
   ```
   Lists which UI surfaces this spec affects. Used by `/verify` and `/qa` to determine which surfaces need testing.

3. Strengthen `**Code:**` guidance: must be relative path prefixes that can be matched by grep. Examples:
   - Good: `[src/PPDS.Dataverse/Services/](../src/PPDS.Dataverse/Services/)`
   - Good: `[src/PPDS.Cli/Commands/PluginTraces/](../src/PPDS.Cli/Commands/PluginTraces/)`
   - Bad: `Multiple — see per-panel sections`
   - Bad: `None` (when code exists)

**New section — Changelog:**

Replace the `**Version:**` frontmatter line with a `## Changelog` section at the bottom of the spec (before Roadmap if present):

```markdown
## Changelog

| Date | Change |
|------|--------|
| 2026-03-18 | Initial spec |
| 2026-03-20 | Added MCP surface section, updated AC-05 status |
```

This is mechanically appendable by `/spec` during reconciliation. Version numbers that nobody bumps are replaced with a dated record of what actually changed.

**Surface sections within specs:**

When a spec covers a domain concept implemented across multiple surfaces, the Specification section should include surface subsections:

```markdown
### CLI Surface
{CLI-specific commands, flags, output format}

### TUI Surface
{Screen layout, keyboard navigation, filter dialogs}

### Extension Surface
{Panel wiring, webview behavior, VS Code integration}

### MCP Surface
{Tool schema, response format}
```

Only include sections for surfaces where the implementation has meaningful surface-specific behavior beyond what the Application Service provides.

### README Auto-Generation

**`specs/README.md` becomes an auto-generated file.** It is never hand-edited. The generation mechanism scrapes frontmatter from all `specs/*.md` files (excluding `CONSTITUTION.md`, `SPEC-TEMPLATE.md`, and `README.md` itself).

**Generated format:**

```markdown
# Specifications

> **Constitution:** All specs must comply with [CONSTITUTION.md](./CONSTITUTION.md).
> See Spec Laws (SL1–SL5) for naming, lifecycle, and granularity rules.

<!-- AUTO-GENERATED — do not edit. Regenerated by /spec. -->

| Spec | Status | Surfaces | Code | Purpose |
|------|--------|----------|------|---------|
| [authentication.md](./authentication.md) | Implemented | All | `src/PPDS.Auth/` | Authentication, credential management, and discovery |
| [plugin-traces.md](./plugin-traces.md) | Implemented | CLI, TUI, Extension, MCP | `src/PPDS.Dataverse/Services/` | Plugin trace querying, inspection, and management |
| ... | ... | ... | ... | ... |
```

**Generation trigger:** `/spec` regenerates the README after every spec create, update, or reconcile operation. The README is also regenerable on demand via a standalone script (`scripts/generate-spec-readme.py` or equivalent).

**Frontmatter scraping rules:**
- **Spec name:** filename (linked)
- **Status:** from `**Status:**` line
- **Surfaces:** from `**Surfaces:**` line (new field)
- **Code:** from `**Code:**` line, shortened to path prefix only
- **Purpose:** first sentence of the Overview section (after the H1)

### Enhanced /spec — Reconcile Mode

The `/spec` skill gains a third mode alongside its existing create and update modes.

**Mode detection (updated Step 2):**
- Spec does not exist → **Forward mode** (create new spec)
- Spec exists, user requests changes → **Update mode** (existing behavior)
- Spec exists, user wants to align with code → **Reconcile mode** (new)

**Reconcile mode behavior:**

1. Read the spec and extract all ACs, Code paths, and type descriptions
2. Read the code at the spec's `Code:` paths
3. Enumerate public types, methods, and behaviors in the code
4. Compare against existing ACs:
   - **Missing ACs:** Code behavior with no corresponding AC → propose new ACs
   - **Stale ACs:** AC describes behavior the code no longer has → flag for removal
   - **Status drift:** AC references a test that now passes/fails differently → update status
5. **Member-count verification:** Count public types/methods in code, count ACs. Report the delta. If ACs cover less than 90% of public surface area, warn explicitly. This prevents the known LLM summarization problem (capturing ~70-80% and missing the rest).
6. Compare spec prose (Architecture, Core Requirements) against actual code structure
7. Present all proposed changes to user for approval before writing
8. After writing, regenerate README and append to Changelog

**Reconcile mode is invoked by:** `/spec <name> --reconcile` or naturally when `/spec` detects the spec exists and code has diverged significantly.

### Skill Updates

#### Skills migrating from README to frontmatter grep

Six skills currently read `specs/README.md` to map code paths to specs. They migrate to grepping spec frontmatter directly.

**Old pattern (in each skill):**
```
Read specs/README.md — maps code paths to specs
```

**New pattern:**
```
Grep all specs/*.md files for **Code:** lines. Match changed file paths against
code path prefixes in frontmatter. This discovers governing specs without a
central index that can drift.
```

Skills to update: `/spec`, `/spec-audit`, `/gates`, `/review`, `/implement`, `ext-panels`

Additionally, `/implement` has hardcoded code-path-to-spec mappings (e.g., `src/PPDS.Cli/Tui/ → specs/tui.md`). These are removed — frontmatter grep replaces them.

#### /implement — Code: frontmatter updates

When `/implement` creates new code paths for a spec (new directories, new files in new locations), it must update the spec's `**Code:**` frontmatter to include the new paths. This ensures the frontmatter grep continues to work as code evolves.

`/implement` does NOT update AC statuses. AC status is owned by `/gates`, which actually runs the tests and can verify pass/fail.

#### /gates — AC status verification

`/gates` already reads specs for AC verification. Its responsibility is clarified:

- After running tests, check AC status column against test results
- If an AC references a test method and that test now passes, the AC status should be ✅
- If an AC says ✅ but the test fails, flag it as a finding
- `/gates` does not update spec files directly — it reports status discrepancies as gate findings

#### /spec-audit — no changes to scope

`/spec-audit` remains read-only. It diagnoses drift and reports findings. It does not modify specs. The treatment for drift findings is running `/spec` in reconcile mode.

### Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| Spec has `Code: None` (Draft, no implementation yet) | Frontmatter grep returns no match for code paths. Spec is still discoverable by name. `/implement` adds Code: path when implementation begins. |
| Spec has `Code: System-wide` (architecture, governance) | Frontmatter grep does not match specific code paths. These specs are loaded by skill-specific rules (e.g., "always read architecture.md"). |
| Multiple specs govern the same code path | Frontmatter grep returns multiple matches. Skill presents all matches to agent for disambiguation. This is expected for cross-cutting specs like `per-panel-environment-scoping.md`. |
| `/spec` reconcile mode finds 0 drift | Reports "spec is current" and skips writing. Does not append a no-op Changelog entry. |
| New spec created with `/spec` | Forward mode creates spec following template. README is regenerated. Changelog starts with "Initial spec" entry. |

## Design Decisions

### Why Constitution Laws Instead of a Separate Governance Doc?

**Context:** Spec governance rules need to be available to every skill that touches specs — /spec, /spec-audit, /gates, /review, /implement, /design.

**Decision:** Add Spec Laws to the Constitution rather than creating a separate governance document.

**Alternatives considered:**
- Separate `GOVERNANCE.md` file: Requires updating every skill to load an additional file. One more thing to forget.
- README.md as governance doc: Wastes the one filename that gets auto-rendered on GitHub. Nobody would read governance rules there.

**Consequences:**
- Positive: Zero new files to load. Every skill already reads the Constitution. Rules are automatically available everywhere.
- Negative: Constitution grows by ~15 lines. Acceptable for a document that's 60 lines today.

### Why Auto-Generated README Instead of Killing It?

**Context:** Six skills depend on the README for code-to-spec mapping. The README is stale (lists 16 of 32 specs).

**Decision:** Auto-generate the README from spec frontmatter. Skills continue reading it. Generation happens in `/spec`.

**Alternatives considered:**
- Kill README, migrate all skills to frontmatter grep: Tested and confirmed agents DO read the README. Migration works but requires updating 6 skills AND losing the GitHub-rendered landing page. More disruption for marginal benefit.
- Hand-maintained README with better discipline: Already failed — the current README proves manual maintenance drifts.

**Consequences:**
- Positive: Skills require minimal changes (just the discovery mechanism swap). README is always accurate. GitHub landing page preserved.
- Negative: /spec has an additional responsibility (README regeneration). Mitigated by being a mechanical append-to-table operation.

### Why /spec Reconcile Mode Instead of a New Skill?

**Context:** Code→spec reconciliation is needed after implementation, during converge, and for periodic maintenance.

**Decision:** Add reconcile as a mode of `/spec` rather than creating `/spec-reconcile` or enhancing `/spec-audit`.

**Alternatives considered:**
- New `/spec-reconcile` skill: Creates ambiguity — "do I use /spec or /spec-reconcile?" One skill for all spec mutations is simpler.
- Enhance `/spec-audit` to fix drift: Violates /spec-audit's read-only design (Rule 1: "produces findings, never modifies"). Audit diagnoses, /spec treats.

**Consequences:**
- Positive: Clean separation — /spec-audit diagnoses, /spec treats. One mental model for spec mutations.
- Negative: /spec becomes more complex (three modes). Mitigated by clear mode detection in Step 2.

### Why /gates Owns AC Status Instead of /implement?

**Context:** AC statuses (✅ ❌ ⚠️ 🔲) need updating as implementation progresses.

**Decision:** `/gates` owns AC status verification because it runs the tests. `/implement` does not touch AC statuses.

**Alternatives considered:**
- /implement updates AC statuses after implementing: Dangerous — marks ACs as "implemented" before tests exist or pass. Creates false confidence. AC status should reflect test reality, not implementation intent.

**Consequences:**
- Positive: AC status is grounded in test results, not agent self-assessment.
- Negative: AC statuses don't update until /gates runs. Acceptable — /gates is always run before PR.

### Why Flat Directory Instead of Nested?

**Context:** 32 specs (25-35 after restructuring) need organization.

**Decision:** Keep `specs/` flat. No subdirectories.

**Alternatives considered:**
- Nested: `specs/features/`, `specs/core/`, `specs/tools/`: Category arguments waste time ("is query-parity a feature or process?"). 30 files don't need hierarchy. Every reference to `specs/plugin-traces.md` would break.

**Consequences:**
- Positive: Simple. All references remain stable. Grep works without recursion flags.
- Negative: 30+ files in one directory. Manageable with alphabetical naming and the auto-generated README.
